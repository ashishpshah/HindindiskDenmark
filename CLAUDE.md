# CLAUDE.md

## Project Overview

Hind Indisk Restaurant is a full-stack-shaped frontend web application for an authentic Indian restaurant chain operating in Denmark. It covers the complete customer journey: browsing the menu, adding items to cart, placing pickup or delivery orders through a multi-step checkout, booking tables, tracking orders, and managing a user account.

All data is currently client-side: static content lives in `src/data/mock.ts` and user-generated data (orders, reservations, cart, auth) is persisted to `localStorage`. There is no backend API.

---

## Business Domain

- **Restaurant:** Hind Indisk ‚Äî two branches in Aarhus (`Frederiksgade 72, 8000 Aarhus C`) and Copenhagen (`Vesterbrogade 41, 1620 K√∏benhavn V`).
- **Order types:** Delivery (+39 DKK, ~45 min) and Pickup (free, ~20 min).
- **Pricing:** All prices include 25% Danish VAT (moms). Tax is shown as a line item in checkout.
- **Coupons:** `WELCOME10` (10% off subtotal), `FAMILY20` (20% off subtotal), `FREEDELIVERY` (waives delivery fee).
- **Menu:** 13 items across 6 categories ‚Äî Starters, Main Course, Rice, Bread, Desserts, Drinks. Items carry vegetarian and spice-level metadata.
- **Reservations:** Branch, date, time, guest count, contact details ‚Äî saved to `localStorage`, not sent to any backend.
- **Authentication:** Mock only. Login always succeeds after a 500 ms delay; user profile is stored in `localStorage`.
- **Languages:** Danish (`da`, default) and English (`en`), switchable at runtime and persisted to `localStorage`.

## Architecture Overview

**State management** is done exclusively via React Context API ‚Äî no Redux or Zustand:
- `CartContext` ‚Äî cart items (`Record<string, number>`), pricing calculations, coupon state, branch, order type.
- `AuthContext` ‚Äî user session, modal open/close state, login/register/forgot-password flows.
- `I18nContext` ‚Äî active language, `t(key)` translation helper.

**Routing** is file-based via TanStack Router. `src/routeTree.gen.ts` is auto-generated by the router plugin on every `vite dev` or `vite build` ‚Äî never edit it manually.

**SSR** is provided by TanStack Start (Nitro backend). SPA prerendering with link crawling is enabled, so all known routes are pre-rendered at build time. The app is also usable as a pure SPA via the generated `index.html` + `.htaccess`.

**Data flow:**
1. Static reference data (`menuItems`, `branches`, etc.) is imported directly from `src/data/mock.ts`.
2. User-generated data is read from and written to `localStorage` using inline `try/catch` blocks.
3. No HTTP calls are made; `src/lib/api/example.functions.ts` is a placeholder for future server functions.

**Provider nesting** (outermost ‚Üí innermost):
```
QueryClientProvider ‚Üí I18nProvider ‚Üí AuthProvider ‚Üí CartProvider ‚Üí <Outlet />
```

## Development Environment

- **Runtime:** Node.js with ESM (`"type": "module"` in package.json).
- **Path alias:** `@/*` resolves to `src/*` (configured in both `tsconfig.json` and Vite via `vite-tsconfig-paths`).
- **Route tree:** TanStack Router auto-generates `src/routeTree.gen.ts` on every dev/build. Adding or renaming a file under `src/routes/` automatically updates it.
- **No test framework** is configured in this repository.
- **Platform integration:** The project was generated on Lovable.dev. The `@lovable.dev/vite-tanstack-config` preset bundles TanStack Start, React, Tailwind, Nitro, path aliases, and error-reporting plugins. Do not register any of those plugins manually in `vite.config.ts`.

## Deployment Overview

The production build targets **Cloudflare Workers** via Nitro. The `postbuild.js` script runs automatically after `vite build` and:
1. Renames `dist/client/_shell.html` ‚Üí `dist/client/index.html` (TanStack Start SPA shell naming convention).
2. Writes `dist/client/.htaccess` with Apache mod_rewrite rules for SPA fallback routing.

SPA prerendering is enabled with `crawlLinks: true`, so all statically reachable routes are pre-rendered at build time.

The app can also be deployed as a static SPA to any Apache host by uploading the contents of `dist/client/` ‚Äî the generated `.htaccess` handles client-side route fallback.
